# Community Guidelines

These guidelines describe how we collaborate in StarkMindsHQ/StrellerMinds-SmartContracts, where to engage, and how we keep the project healthy.

## Where to Engage

- Issues: bugs, feature requests, and concrete tasks
- Pull requests: proposed changes to code and documentation
- Discussions: Q&A, ideas, and design exploration

## How We Work

### Be Respectful and Professional

All community participation is governed by [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md).

### Prefer Written Context

To keep work reviewable and searchable:
- Use issues for decisions, trade-offs, and follow-ups
- Link related issues/PRs and reference relevant docs
- Include reproduction steps and environment details for bugs

### Security and Responsible Disclosure

If you believe you have found a security issue:
- Do not open a public issue with exploit details
- Contact the maintainers listed in [.github/GOVERNANCE.md](.github/GOVERNANCE.md) to coordinate a fix

## Contribution Policies

Contribution expectations are documented in:
- [CONTRIBUTING.md](CONTRIBUTING.md) for the short version
- [docs/contributing.md](docs/contributing.md) for the detailed process

Common expectations:
- All work is tracked by an issue
- Changes include tests where applicable
- Contract interface/event changes include documentation updates
- Code quality checks (formatting and clippy) are run before requesting review

## Community Processes

### Triage and Prioritization

The project uses a lightweight triage process described in [.github/GOVERNANCE.md](.github/GOVERNANCE.md), including labels and priority levels.

### Reviews and Decision Making

- CODEOWNERS drives reviewer assignment
- Maintainers may request design discussion before large changes land
- Security-sensitive changes receive additional scrutiny

### Moderation

Maintainers may moderate issues, PRs, and discussions to:
- keep conversations productive and on-topic
- remove content that violates the code of conduct
- request rephrasing or additional context when needed

## Engagement Cadence

Suggested minimum cadence for maintainers:
- Weekly: issue triage and response to new questions
- Monthly: review open PR queue health and stale issues
- Quarterly: community health review and process adjustments

## Community Health Review

Use this checklist quarterly:
- Response time to new issues and PRs
- Number of stale issues and PRs, and the main causes
- CI failure themes (clippy/fmt/tests) and recurring contributor friction
- Documentation gaps reported by contributors
- Onboarding outcomes using [docs/tool-effectiveness-review.md](docs/tool-effectiveness-review.md)
